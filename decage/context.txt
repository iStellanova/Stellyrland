**Role:** You are an expert Python developer and an advanced Gentoo Linux system administrator.

**Context:**
We are extending a Gentoo declarative package manager called `decage` (Declarative Portage). The base `decage.py` script is already written and successfully handles synchronizing the Gentoo `@world` set using the native `portage` API, including installing, deselecting, and running `--depclean`.

We now need to implement a modular Plugin Architecture and a File/Directory management system, porting over logic from a previous Artix Linux tool.

**Requirements:**

**1. Integrate Plugin Architecture into `decage.py`:**
- Update the main execution loop in `decage.py` to parse additional variables from the user's `source.py` file: `execution_order` (a list of strings), `plugins` (a dictionary mapping strings to instantiated plugin objects), `files` (a dict of File objects), and `directories` (a dict of Directory objects).
- Create a base `Plugin` class that requires an `apply(self, store, dry_run=False)` method.
- During execution, `decage` should iterate through `execution_order`. If a step matches the portage sync (e.g., "portage"), it runs the existing emerge logic. Otherwise, it looks up the step in the `plugins` dictionary and calls its `apply()` method.

**2. File & Directory Management:**
- Create the base `File` and `Directory` classes.
- **Logic:** These should accept a `source_file`/`source_directory` (relative to the user's configuration repo) and copy or symlink them to the target destination (e.g., `~/.config/...`).
- **Permissions:** Implement logic to enforce ownership (`chown`, using an imported `USER` constant) and permissions (`chmod`, defaulting to `0644` for files and `0755` for directories). Ensure this is integrated as a default plugin step.

**3. The Gentoo Overlay Plugin (Replacing AUR logic):**
- Create a `RepositoryPlugin` to manage Gentoo overlays.
- **Input:** A list or set of repository names (e.g., `{"guru", "steam-overlay"}`).
- **Action:** Check currently enabled overlays using `eselect repository list -i`. If a requested overlay is missing, run `eselect repository enable <repo_name>`. If a new repo is added, automatically run `emaint sync -r <repo_name>` to fetch it.

**4. Translating Arch Plugins to Gentoo:**
- **CacheCleanupPlugin:** Create a plugin that runs `eclean-dist -d` and `eclean-pkg -d` to clear old source files and binary packages.
- **SyncPlugin:** Create a plugin that runs `emaint sync -A` to update the main Portage tree and all overlays.
- *(Note: Do not build an Orphan Cleanup plugin. This is already handled natively by `decage.py`'s `--depclean` flag).*

**5. Distro-Agnostic Plugins:**
Create the following plugins. The underlying CLI commands are distro-agnostic and should execute via `subprocess`:
- `OpenRCPlugin` (manages `rc-update add <service> <runlevel>` and `rc-update del <service>`)
- `GsettingsPlugin` (manages `gsettings set/get`)
- `SnapperPlugin` (runs `snapper create ...`)
- `GrubPlugin` (runs `grub-mkconfig -o /boot/grub/grub.cfg`)
- `BtrfsMaintenancePlugin` (runs `btrfs balance` and `btrfs scrub`)
- `XdgDirsPlugin` (runs `xdg-user-dirs-update`)

**Output:**
Please generate the Python code for the `Plugin`, `File`, and `Directory` classes, as well as the specific plugin implementations. Provide instructions on how to structure these within the `decage` module (e.g., placing them in a `plugins.py` file) and show exactly how to modify the existing `decage.py` main loop to trigger them while preserving the existing ANSI formatting and error handling.
